# Quantmind — App Rules & Operational Guidelines

> These rules govern how Quantmind behaves across all surfaces: mobile app, web dashboard, AI layer, simulation engine, and backend infrastructure. They exist to ensure the platform remains **strict in its standards, transparent in its operations, and professional in every user interaction.**

---

## 1. Financial Integrity Rules

### 1.1 No Investment Advice
- Quantmind is an **educational and analytical simulation tool**, not a licensed financial advisor.
- The AI assistant, simulation results, and all risk metrics must **never recommend specific securities to buy, sell, or hold**.
- Every output that touches financial outcomes must carry a contextual disclaimer where appropriate.
- The phrase *"This is not financial advice"* must appear in PDF reports, onboarding, and the AI assistant's first session message.

### 1.2 Simulation Honesty
- Simulations must **never be presented as predictions**. They are probability distributions of possible outcomes.
- All charts, metrics, and result screens must clearly label outputs as *"simulated"* or *"projected"* — never *"expected"* without qualification.
- Confidence intervals must display the full range, including unfavourable tails. Truncating downside scenarios is prohibited.
- The number of simulation paths used must always be visible to the user on the results screen.

### 1.3 Methodology Transparency
- Every risk metric (VaR, CVaR, Sharpe, etc.) must have an accessible plain-language explanation reachable within one tap.
- Model assumptions (GBM drift, constant volatility, normal distribution of returns) must be disclosed to users accessing advanced features.
- Backtesting results, where displayed, must include the time period, asset universe, and any survivorship bias disclaimers.
- When advanced models (jump diffusion, fat tails, regime-switching) are active, the user must be informed which model is running.

### 1.4 Data Accuracy
- Market data sourced from third-party APIs must display the **source name and timestamp** of last fetch.
- Stale data (older than 24 hours for daily inputs) must trigger a visible staleness warning before simulation is run.
- Manual user inputs (expected return, volatility, correlations) must be validated against plausible ranges; inputs outside those ranges must surface a non-blocking warning, not a silent override.

## 2. AI Behaviour Rules

### 2.1 Scope of AI Responses
- The AI assistant must **stay within the domain** of portfolio risk analysis, quantitative finance concepts, and Quantmind feature guidance.
- Requests for specific stock picks, market timing calls, or personalised investment plans must be gracefully declined with a redirect to the simulation tools.
- The assistant must not speculate on future market prices or claim predictive capability.

### 2.2 Model Transparency
- When the AI provides an analysis, it must be attributable to a specific prompt workflow (e.g. *"Portfolio Doctor analysis"*, *"VaR Explanation"*) — not presented as a generic oracle.
- If a response was generated by a lower-capability model tier (Haiku vs Opus), the depth of analysis must be accurate to that model's capability — no false depth signals.
- Users must be able to request a plain-language explanation of *how* any AI conclusion was reached.

### 2.3 Limitations Disclosure
- The AI must proactively acknowledge uncertainty when discussing tail risk, extreme scenarios, or predictions beyond standard model assumptions.
- If the AI's context window does not include sufficient portfolio data to give a complete answer, it must say so rather than hallucinate assumptions.
- AI responses must never contradict the simulation engine's numerical outputs. If a discrepancy exists, the numerical output takes precedence and the AI must defer to it.

### 2.4 User Expertise Adaptation
- The AI must detect user expertise level through interaction signals and adapt explanation depth accordingly — without being condescending to beginners or oversimplifying for advanced users.
- Jargon must always be accompanied by a brief inline definition on first use within a session.
- The AI must never make users feel uninformed for not understanding a concept. Tone must remain encouraging and neutral.

### 2.5 Feedback Loop Integrity
- Thumbs up / thumbs down feedback on AI responses must be logged accurately and used only to improve prompt quality — never to suppress legitimate criticism of a result.
- Negative feedback must not alter the underlying simulation or metric outputs.

## 3. Subscription & Entitlement Rules

### 3.1 Honest Feature Gating
- Features behind a paywall must be clearly labelled as such **before** a user attempts to use them — not after they have invested time configuring an input.
- Upgrade prompts must be **informative, not manipulative**. They must state the specific feature being unlocked and the price — no dark patterns, artificial urgency, or misleading trial terms.
- Trial limits (7 days, no card required) must be enforced exactly as advertised. No silent auto-charges.

### 3.2 Tier Enforcement
- Subscription tier entitlements must be enforced **server-side** on every API call. Client-side tier checks are for UI display only and must never be the sole access gate.
- A user downgraded from Pro to Plus must immediately lose access to Pro-only features, with their data preserved and accessible again upon upgrade.
- Free tier simulation path limits (2,000) must be enforced at the simulation engine level — not merely hidden in the UI.

### 3.3 Pricing Transparency
- Prices displayed in-app must match App Store / Google Play pricing exactly at all times.
- Annual pricing savings must be calculated accurately and displayed as both the percentage saved and the absolute dollar amount.
- Student tier verification must use a third-party service (SheerID / UNiDAYS / institutional email) — self-declaration is not sufficient.

## 4. Data Privacy & Security Rules

### 4.1 Data Minimisation
- Quantmind must collect only the data necessary to deliver the service. No behavioural data must be sold or shared with advertisers.
- Portfolio data (assets, allocations, correlations) is user-private by default and must never be visible to other users.
- Anonymised, aggregated analytics data used for product improvement must be non-identifiable at the record level.

### 4.2 Prompt & Context Security
- User portfolio data injected into AI prompts must be stripped of any personally identifiable information (name, email, account ID) before transmission to the model.
- AI prompt logs stored in analytics_events must not contain raw portfolio values — only metadata (task type, token counts, latency, tier).
- The Anthropic API key must be rotated monthly and must never appear in client-side code, logs, or error messages.

### 4.3 Authentication & Access
- All API routes that access user data must validate the JWT token and confirm the authenticated user matches the requested resource.
- Admin and dashboard routes must be restricted to email-allowlisted accounts only. No public registration path for admin access.
- Impersonation mode (view app as a user) must be read-only, require explicit confirmation, and generate an immutable audit log entry.

### 4.4 Regulatory Compliance
- GDPR: Users in the EU must be able to export all their data and request deletion within 30 days of a request.
- CCPA: California users must be informed of data collection practices at sign-up and have an opt-out path for any non-essential data use.
- All data deletion requests must cascade through portfolios, simulations, analytics events, and AI query logs.

## 5. Simulation Engine Rules

### 5.1 Numerical Accuracy
- All risk metric calculations must be validated against known textbook reference values in the test suite before any release.
- VaR and CVaR must be computed from the full empirical simulation distribution — not approximated from a parametric normal assumption unless explicitly disclosed.
- Correlation matrix inputs must be validated for positive semi-definiteness before Cholesky decomposition. Invalid matrices must be rejected with an informative error, not silently altered.

### 5.2 Reproducibility
- Every simulation result must store the random seed used so that results can be exactly reproduced from the stored parameters.
- Cached simulation results must be clearly marked as cached (with the original run timestamp) when served from the cache rather than freshly computed.
- Any change to the simulation engine's core algorithms constitutes a version increment and must be logged in the changelog.

### 5.3 Performance Contracts
- 2,000-path simulations must complete in under 3 seconds. 10,000-path simulations must complete in under 10 seconds under normal load.
- If a simulation job times out, the user must receive a clear, honest error message — not a spinner that runs indefinitely.
- Simulation results must never be partially returned. A result is either complete and validated, or the job is marked as failed.

## 6. UI & Communication Rules

### 6.1 Clarity Over Cleverness
- Risk metrics must be displayed with their full name, abbreviation, and a one-sentence definition visible on the results screen without requiring navigation.
- Probability values must always be shown as percentages with sufficient context (e.g. *"28% chance of a loss greater than 10% over 1 year"* — not just *"28%"*).
- Colour coding (green / yellow / orange / red) for risk levels must be accompanied by a text label. Colour must never be the sole indicator of severity.

### 6.2 Error & Edge Case Communication
- Error messages must be written in plain language, explain what went wrong, and suggest a corrective action. Error codes alone are not acceptable user-facing messages.
- Empty states (no portfolios, no simulations yet) must guide the user to the next action — not display a blank screen.
- When a simulation produces extreme or unexpected results, a non-alarmist contextual note must explain why (e.g. high input volatility, very long time horizon).

### 6.3 Notification Standards
- Push notifications must be opt-in and categorised (risk alerts, product updates, goal milestones). Users must be able to disable each category independently.
- Risk alert notifications must be actionable — they must link directly to the relevant portfolio or metric, not the app home screen.
- Marketing notifications must be clearly distinguishable from risk or system alerts. They must not use urgency language that mimics a genuine risk event.

## 7. Web Dashboard & OTA Update Rules

### 7.1 Update Safety
- OTA updates must follow a staged rollout: 5% → 25% → 50% → 100%. No update may jump directly to full rollout.
- If the crash rate for any active cohort exceeds the configured threshold within 15 minutes of an OTA push, the rollout must auto-pause and alert the responsible engineer.
- Every OTA publish must require a human-authored release note. Blank or placeholder notes are rejected by the CI pipeline.

### 7.2 Feature Flag Discipline
- Every feature flag change must include a written reason. The audit log is immutable.
- A/B model experiments (e.g. Haiku vs Sonnet for a given task) must define a success metric and a planned end date before launch.
- Emergency kill switches must be tested quarterly to confirm they function correctly in production.

### 7.3 Prompt Governance
- Prompt templates are immutable once published. Changes require creating a new version, not editing the live template.
- Every published prompt must have at least one reviewer sign-off before it reaches production.
- Prompts must be tested in the playground against at least three representative portfolio scenarios before publishing.

## 8. Monitoring & Incident Response Rules

### 8.1 Observability Requirements
- App crash rate, API error rate, simulation failure rate, and AI response latency must be monitored in real time with alerting thresholds configured for each.
- Any P1 incident (service unavailable, data loss, security breach) must have a post-mortem published internally within 5 business days.
- All monitoring dashboards must be accessible to the on-call engineer without requiring another team member to grant access.

### 8.2 Incident Communication
- If a service degradation affects user-facing functionality, a status update must be posted within 30 minutes of confirmed impact.
- Users who initiated a simulation or AI request during an incident must be notified to retry — results from degraded periods must not be silently served.
- Security incidents involving potential user data exposure must be disclosed to affected users within the timeframes required by GDPR and CCPA.

## 9. Content & Legal Rules

### 9.1 Disclaimers
The following disclaimer must appear in the app onboarding, PDF reports, and the AI assistant's first session message:

> *Quantmind provides educational simulation tools for informational purposes only. Simulated results do not guarantee future outcomes. All investments carry risk, including potential loss of principal. This is not financial advice. Consult a qualified financial advisor for personalised guidance.*

### 9.2 Academic & Methodology Citations
- All quantitative methods used (GBM, Cholesky decomposition, VaR, CVaR, jump diffusion, regime-switching) must be citable to peer-reviewed academic sources in the public documentation.
- No proprietary "black box" models may be presented to users without a plain-language description of the methodology.

### 9.3 Intellectual Property
- All third-party data providers (Yahoo Finance, Alpha Vantage, IEX Cloud) must be attributed per their terms of service.
- User-generated content (portfolio configurations, simulation parameters) remains the property of the user. Quantmind may use anonymised, aggregated versions for product improvement only.

## 10. Password & Authentication Strength Rules

### 10.1 Password Requirements
All passwords created or updated within Quantmind must meet the following minimum standards, enforced on both the client (for immediate feedback) and the server (as the authoritative gate):

- **Minimum length:** 12 characters
- **Maximum length:** 128 characters — prevents bcrypt denial-of-service via oversized inputs
- **Required character classes:** at least one uppercase letter, one lowercase letter, one number, and one special character (`!@#$%^&*()_+-=[]{}|;:,.<>?`)
- **Prohibited patterns:** password must not contain the user's email address, username, or the string "quantmind" (case-insensitive)
- **Common password block:** passwords must be checked against a list of the 10,000 most common passwords and rejected with the message: *"This password is too common. Please choose something more unique."*
- **Breached password check:** new passwords must be checked against the HaveIBeenPwned Pwned Passwords API (k-anonymity model — only the first 5 characters of the SHA-1 hash are transmitted) and rejected if found in a known data breach, with the message: *"This password has appeared in a known data breach. Please choose a different one."*

### 10.2 Real-Time Password Strength Indicator
The sign-up and password change screens must display a live strength meter that gives users meaningful, actionable feedback — not just a colour bar:

- **Four levels:** Weak / Fair / Strong / Very Strong, each accompanied by a plain-language improvement hint (e.g. *"Try mixing character types"* or *"Add more length"*)
- The **Submit / Create Account** button must remain disabled until the password reaches at least **Strong**
- Strength assessment must run entirely client-side in real time using a library such as `zxcvbn` — no server round-trip per keystroke
- The meter must not expose which specific rule failed in a way that helps an attacker map the validation logic. Hints must be directional, not diagnostic

### 10.3 Password Change & Reset Controls
- Users must re-enter their **current password** to set a new one (except when arriving via a valid reset link)
- After a successful password change, all **other** active sessions for that account must be invalidated immediately. The current session remains valid
- Password reset links delivered by email must expire in **1 hour** and be invalidated after a single successful use
- After **5 consecutive failed login attempts**, the account must enter a **15-minute lockout**. The lockout response must not reveal whether the email exists in the system
- Locked accounts must display: *"Too many attempts. Please wait 15 minutes or reset your password."*
- Repeated lockouts for the same email within 24 hours must trigger an internal alert as a potential account takeover signal

### 10.4 Multi-Factor Authentication
- TOTP-based MFA (authenticator app) must be **available** to all paid users (Plus, Pro, Student) and **mandatory** for all admin and dashboard accounts
- MFA enrolment must be prominently offered during onboarding for paid tiers — not buried in settings
- Backup codes (8 single-use codes) must be generated at MFA enrolment and presented once. They must be stored as hashed values — never in plain text
- Admin and dashboard users without MFA enrolled must be blocked from protected routes and redirected to the MFA setup screen on every login until enrolment is complete
- TOTP codes use a **30-second validity window** with a ±1 step tolerance to accommodate minor clock drift
- MFA cannot be disabled for admin accounts, even by the account owner — only by another admin with documented justification

### 10.5 Session & Credential Hygiene
- Passwords must be hashed using **bcrypt** with a minimum cost factor of 12. Supabase Auth handles this by default — do not override with a lower value
- Passwords must never appear in logs, analytics events, error messages, support tickets, or any other stored record
- The persistent "stay signed in" option must not extend sessions beyond **30 days** regardless of refresh activity
- Users must be able to view all active sessions (device type, approximate location, last active timestamp) from their profile settings, and revoke any individual session without affecting others
- Sign-out must revoke the refresh token server-side — not just clear local storage on the device

## 11. Privacy Rules

### 11.1 What Quantmind Collects — and What It Does Not

| Data Type | Collected | Purpose | Retention |
|---|---|---|---|
| Email address | Yes | Authentication, transactional email | Until account deletion |
| Portfolio configurations | Yes | Core simulation feature | Until deleted by user |
| Simulation results | Yes (cached) | Performance, history | 90 days, then purged |
| AI query metadata | Yes (anonymised) | Cost analytics, quality improvement | 12 months |
| AI message content | No | — | Not stored |
| Device type / OS | Yes | Crash reporting (Sentry) | 30 days |
| IP address | Yes (rate limiting only) | Abuse prevention | 24 hours, then purged |
| Precise location | No | — | Not collected |
| Browsing or behavioural data | No | — | Not collected |
| Payment card details | No | Handled entirely by RevenueCat / App Stores | Not stored by Quantmind |
| Third-party advertising identifiers | No | — | Not collected |

### 11.2 Consent & Transparency
- A clear, plain-language **Privacy Policy** must be linked from the sign-up screen, the app settings, and the web footer
- The Privacy Policy must be written at a reading level accessible to a general audience. Legal terms must be accompanied by plain-language explanations
- Users must explicitly consent to the Privacy Policy during sign-up via a non-pre-ticked checkbox. Proceeding without consent must be blocked
- Any material change to data collected, retention periods, or third-party sharing must be communicated to existing users by email at least **14 days before** the change takes effect, with the option to delete their account before it applies
- The app must never use dark patterns to obscure privacy choices. Consent controls must be equally easy to deactivate as to activate

### 11.3 User Rights & Data Control
Users must be able to exercise the following rights directly from the app settings screen, without needing to contact support:

- **View their data:** a structured summary of all data Quantmind holds — account info, portfolios, simulation history, subscription history
- **Export their data:** a downloadable JSON export of all personal data, delivered within 48 hours of request
- **Delete their account:** permanent deletion of all personal data across every table, completed within 30 days, with a confirmation email when done
- **Withdraw analytics consent:** users may opt out of non-essential analytics at any time from settings; this must take immediate effect and purge previously collected non-essential analytics linked to their account

Account deletion must cascade in the following order and must not be considered complete until every step succeeds:

`simulations` → `portfolios` → `analytics_events` → `support_tickets` → `support_messages` → `webhook_events` → `users`

### 11.4 Third-Party Data Sharing
Quantmind shares data with the following parties **only**, for the stated purpose, with no resale or secondary use permitted:

| Third Party | Data Shared | Purpose |
|---|---|---|
| Supabase | All user data (stored) | Database and authentication infrastructure |
| Anthropic | Anonymised portfolio context (transient) | AI analysis — not stored by Anthropic per API terms |
| RevenueCat | User ID, subscription events | Subscription management |
| Sentry | Anonymised error context, no PII | Crash reporting |
| Expo / EAS | App metadata | OTA update delivery |
| Resend | Email address | Transactional email delivery |

No data is shared with advertising networks, data brokers, or any party not listed above. This list must be reviewed quarterly and kept current in the Privacy Policy.

### 11.5 Children's Privacy
- Quantmind is not directed at children under the age of 13 (or 16 in the EU under GDPR)
- The sign-up flow must collect date of birth and block account creation for users below the minimum age, with the message: *"Quantmind is for users aged 13 and over."*
- If an underage account is identified post-signup, it must be suspended immediately and all associated data deleted within 7 days
- No data from underage users may be retained for any purpose, including anonymised analytics

### 11.6 Cookie & Tracking Policy (Web)
- The public marketing site (`quantmind.app`) must display a cookie consent banner on first visit for users in the EU, UK, and California
- **Strictly necessary cookies** (session, CSRF token) are set without consent as they are required for the service to function
- **Analytics cookies** (e.g. Google Analytics) require explicit opt-in and must not fire before consent is given
- **No advertising or third-party tracking cookies** are permitted on any Quantmind web property
- Users must be able to withdraw cookie consent at any time via a clearly visible link in the footer
- The admin and dashboard apps are internal tools exempt from cookie banners, but the no-advertising-cookies rule applies equally

### 11.7 AI & Privacy
- The AI assistant must not ask users for personally identifiable information (full name, address, phone number, tax or account details) in the course of providing portfolio analysis
- Portfolio context sent to the Anthropic API must contain only asset tickers, weights, and numerical metrics — no names, email addresses, or account identifiers
- AI conversation history is held in-memory for the session only and must not be persisted to the database after the session ends
- The Privacy Policy must explicitly state that AI interactions are processed by Anthropic and must link to Anthropic's API data usage policy

## 12. General Conduct Rules

| Rule | Standard |
|---|---|
| Honesty about limitations | The app must never overstate its predictive accuracy or imply certainty it does not have |
| Consistency | Rules applied to one user or cohort must apply equally to all equivalent users and cohorts |
| No dark patterns | No misleading UI, no hidden charges, no pre-ticked consent boxes, no fake countdown timers |
| Accessibility | Core functionality must meet WCAG 2.1 AA standards; colour contrast, screen reader labels, and font scaling are non-negotiable |
| Versioning | Every change to the simulation engine, AI prompts, or risk metric calculations constitutes a versioned release logged in the public changelog |
| Audit trail | All admin actions (tier changes, user suspensions, OTA pushes, flag changes) must generate an immutable audit log entry with author, timestamp, and reason |
